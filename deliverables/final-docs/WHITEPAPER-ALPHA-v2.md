# ALPHA Whitepaper v2

Status: Simulation-backed revision draft  
Date: 2026-04-16  
Document type: Phase 1 internal whitepaper draft  
Simulation basis: one approved snapshot and four standard completed scenarios only  
Related basis memo: [SIMULATION-BASIS-ALPHA-v2.md](./SIMULATION-BASIS-ALPHA-v2.md)

## 1. Executive Summary

`ALPHA` should still be positioned as the internal rights and settlement layer for the BGC x iBLOOMING ecosystem.

However, Whitepaper v2 should not claim that the final pilot policy has already been selected from the current standard scenario set.

The current local simulation basis uses:

- the latest approved canonical snapshot: `BGC canonical bundle v3 with DATA_AGG`
- four standard scenarios: `Baseline`, `Conservative`, `Growth`, and `Stress`

The conclusion is clear:

- the strategic direction remains valid
- the token design mechanics remain documentable
- but none of the four standard scenarios is yet founder-ready as a production pilot policy because all four remain `risky` on reward concentration

## 2. Problem Statement

The BGC x iBLOOMING business already generates real economic activity through:

- fiat inflow
- `Purchase Credit (PC)`
- `Sales Point (SP)`
- rewards
- payouts
- digital product activity

The structural problem is not a lack of value creation.

The structural problem is that value still moves through fragmented loops rather than one unified internal settlement design. That fragmentation limits:

- value circulation inside the ecosystem
- cross-service utility
- policy visibility
- and disciplined control over how value exits the system

## 3. Phase 1 Positioning

Whitepaper v2 should keep the same core Phase 1 positioning:

- `ALPHA` is an internal settlement and rights layer
- it is policy-governed, not market-priced
- it is tied to existing business activity rather than speculative issuance
- it is intended to structure value before any public-token path is considered

Phase 1 still does **not** imply:

- public token trading
- open transferability
- live onchain market behavior
- automatic conversion into later public assets such as `iBC` or `iBTC`

## 4. Simulation Method For v2

Whitepaper v2 should explicitly state that its policy interpretation is based on:

- one approved local dataset only

## 5. What The Current Simulation Proves

### Supported strongly enough for Whitepaper claims

The current comparison supports these Phase 1 claims:

- `ALPHA` remains a credible internal settlement direction for existing BGC x iBLOOMING economic flows.
- The treasury side remains manageable across all four standard scenarios under the current model.
- Affiliate performance remains strong across the full scenario set.
- Active-user support remains strong across the full scenario set.
- Controlled policy levers such as caps, sink target, fees, and cash-out windows materially shape issuance, hold, spend, and cash-out behavior.

### Not supported strongly enough for final pilot-policy claims

The current comparison does **not** support these claims yet:

- that one of the four standard scenarios is already safe enough to lock as the default pilot policy
- that fairness concentration is already solved
- that operating-cost reduction is proven
- that tax optimization is proven
- that the current standard sink designs already imply mature internal utility demand

## 7. The Whitepaper Boundary v2 Must Respect

Whitepaper v2 should finalize the strategic narrative, but not overstate policy finality.

### What Whitepaper v2 can lock

- the Phase 1 purpose of `ALPHA`
- the role of `PC` and `SP` as economic source signals
- the need for controlled internal issuance and controlled release
- treasury and fairness as hard decision gates
- milestone-based governance over policy changes

### What Whitepaper v2 should leave explicitly open

- the final production parameter set
- the approved fairness-control policy
- the final cash-out posture in production
- the full sink scope for Phase 1
- legal and compliance treatment for later expansion

## 8. Strategic Objective Framing For Whitepaper v2

Whitepaper v2 should use more precise claim language:

### Strongest current support

- `Affiliate Acquisition`: strong across all four standard scenarios
- `Active-User Growth`: strong across all four standard scenarios

### Partial or scenario-dependent support

- `Revenue Growth`: stronger in `Baseline` and `Growth`, softer in `Conservative` and `Stress`

### Not yet proven enough

- `Operational Cost Reduction`
- `Tax Optimization`

This matters because Whitepaper v2 should read as an evidence-based policy document, not as a marketing statement.

## 9. Required Founder Decision Gates

Whitepaper v2 should formalize the gates that must be satisfied before policy lock:

- payout / inflow ratio must stay below the model threshold
- reserve runway must stay above the model threshold
- top-10% reward concentration must fall below the model threshold
- cash-out policy must remain explicitly governed
- pilot parameters must be milestone-reviewed rather than assumed fixed forever

In the current four-scenario comparison, the unresolved gate is concentration.

## 10. Whitepaper v2 Conclusion

Whitepaper v2 should conclude with a disciplined position:

`ALPHA is ready to be documented as the internal settlement architecture for Phase 1, but the current four standard scenarios are not yet sufficient to lock the final pilot policy because reward concentration remains above the model threshold in every case.`

That conclusion is stronger and more defensible than pretending the standard scenario set has already solved the policy problem.
